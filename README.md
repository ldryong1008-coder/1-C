# Mini NPU Simulator

본 프로젝트는 Neural Processing Unit(NPU)의 핵심 연산인 MAC(Multiply-Accumulate) 연산을 Python으로 구현한 시뮬레이터입니다. 입력된 패턴을 분석하여 특정 필터(십자가, X)와의 유사도를 판별합니다.

## 2. 핵심 구현 및 로직 설명 (실제 코드 포함)

### [평가기준 5] 라벨 정규화 (Label Normalization) 구현
```python
def normalize_label(label):
    label_str = str(label).lower()
    if label_str in ['+', 'cross']:
        return 'Cross'
    if label_str in ['x']:
        return 'X'
    return str(label)
```
**설명**: JSON 파일 내 `expected` 값이나 `filter` 키는 작성자에 따라 `+`, `x`, `cross`, `X` 등 다양하게 표기될 수 있습니다. 내부에서 조건문을 활용해 비교하기 쉽도록 모든 라벨을 대소문자 구분 없이 처리하여 표준화된 문자열 `Cross` 와 `X`로 통일하는 로직입니다.

### [평가기준 10, 11] MAC 연산 로직 및 로직 설계
```python
def mac_operation(pattern, filter_matrix):
    score = 0.0
    rows = len(pattern)
    if rows == 0: return score
    cols = len(pattern[0])
    for i in range(rows):
        for j in range(cols):
            score += float(pattern[i][j]) * float(filter_matrix[i][j])
    return score
```
**설명**: 외부 라이브러리(NumPy)를 사용하지 않고 파이썬 표준 문법을 활용했습니다. 입력된 `pattern`과 `filter_matrix`의 행(`rows`)과 열(`cols`) 크기를 가져온 뒤, 중첩된 `for` 루프를 순회하며 동일한 인덱스(`[i][j]`)에 위치한 값을 곱하고(`Multiply`) 그 결과값을 `score` 변수에 누적하여 더합니다(`Accumulate`). 이것이 MAC 연산의 본질적인 구조입니다.

### [평가기준 12] 모듈 분리 이유
프로그램은 데이터 입출력(UI), 정규화(`normalize_label`), 핵심 연산(`mac_operation`), 성능 측정(`measure_mac_time`) 단위로 함수가 완전히 분리되어 있습니다.
**이유**: 단일 책임 원칙(SRP)을 지키기 위함입니다. 핵심 연산 로직과 콘솔 입출력을 분리하면 모드 1과 모드 2에서 `mac_operation`을 재사용할 수 있으며, 추후 단위 테스트 작성과 알고리즘 교체(예: NumPy 도입) 시 유지보수가 매우 유리해집니다.

### [평가기준 13] 시간 측정 경계 설정
```python
def measure_mac_time(pattern, filter_matrix, repeat=10):
    start = time.perf_counter()
    for _ in range(repeat):
        mac_operation(pattern, filter_matrix)
    end = time.perf_counter()
    return ((end - start) * 1000) / repeat
```
**설명**: `time.perf_counter()`를 사용하여 가장 높은 정밀도의 시간을 측정합니다. 파일 입출력(I/O)이나 화면 출력(`print`) 코드는 절대 시간 측정 경계 안에 포함시키지 않고, 오직 순수한 `mac_operation` 호출만을 `repeat` 횟수만큼 반복하도록 경계를 설정하여 알고리즘 자체의 순수 연산 속도를 보장합니다.

## 3. 심층 답변 (인터뷰 / 기술 설명)

### [평가기준 14] MAC 연산의 핵심 기술 원리
MAC 연산은 두 행렬의 원소별 곱셈과 전체 합성을 수행하는 작업으로, 딥러닝(특히 CNN 모델)에서 특정 필터(가중치)가 입력 데이터(이미지 등)에서 얼마나 강한 특징(Feature)을 지니고 있는지 '유사도'를 구하는 합성곱(Convolution) 연산의 가장 밑바탕이 됩니다. AI 칩(NPU)은 이 단순 반복 연산을 병렬로 수천 개씩 동시 처리할 수 있도록 최적화된 하드웨어입니다.

### [평가기준 15] 시간 복잡도 O(N²) 경향 원리적 설명
패턴 행렬의 한 변의 길이를 `N`이라고 할 때, 2차원 행렬 전체의 원소 개수는 `N * N = N²`개가 됩니다.
위에서 구현한 MAC 코드는 바깥쪽 `for i`가 N번 반복되고, 그 안의 `for j`가 N번 반복되므로 곱셈과 덧셈 연산이 정확히 N²번 수행됩니다. N이 5에서 25로 5배 늘어나면 연산 횟수는 25에서 625로 25배 늘어나기 때문에, 시간 복잡도는 O(N²)으로 귀결됩니다. (로그의 성능 표에서도 측정 시간이 대략 25배 증가함을 증명하고 있습니다.)

### [평가기준 16] 부동소수점 오차 원인과 Epsilon의 필요성
컴퓨터는 무한한 실수를 유한한 메모리(IEEE 754 표준 2진수 부동소수점)에 저장하므로, 0.1과 같은 소수를 정확히 표현하지 못하고 미세한 오차가 누적됩니다.
따라서 이론적으로 동일해야 할 두 MAC 점수가 `12.500000000000002`와 `12.5`로 다르게 나타날 수 있습니다. 이를 파이썬의 `==` 연산자로 직접 비교하면 '다르다'고 판정하는 치명적인 버그가 생기므로, 두 값의 차이가 `1e-9` (Epsilon)보다 작으면 같은 값으로 취급하는 허용오차 기반 판별 로직이 필수적입니다.

### [평가기준 17] 동점 처리 규칙 이원화 이유
* **모드 1 (사용자 입력)**: 단순히 A필터와 B필터 중 무엇과 더 유사한지 묻는 것이므로, 동점일 경우 `판정 불가`라고 솔직하게 응답하는 것이 사용자 경험에 맞습니다.
* **모드 2 (JSON 분석)**: 주어진 `expected` (예: "X" 필터 도출)를 반드시 만족해야 하는 테스트 환경입니다. 동점이 나와서 결과가 `UNDECIDED`가 된 경우, 시스템은 요구된 "X" 필터를 확정적으로 선택하지 못했기 때문에, 이는 자동화된 테스트 환경의 기준에 부합하지 않아 `FAIL`로 집계하는 것이 논리적으로 타당합니다.

### [평가기준 18] 확장 계획
현재의 2차원 단일 채널 필터를 넘어, 컬러 이미지(RGB 3채널)를 지원하기 위한 3차원 배열 처리(Tensor 연산)로 확장이 필요합니다. 또한 여러 개의 필터(Filter Bank)를 한 번에 검사할 수 있는 배치(Batch) 처리 구조로 발전시킬 수 있습니다.

### [평가기준 19] 최적화 계획
현재는 2차원 배열의 인덱스(`[i][j]`)에 접근하고 있습니다. 2차원 배열을 1차원 배열(길이 N²)로 Flatten(평탄화)하여 메모리의 물리적 연속성을 높이면 캐시 적중률(Cache Hit Rate)이 극대화되어 연산 속도가 더욱 단축됩니다. (보너스 과제 개념)

### [평가기준 20] 병목 지점 및 해결 방안
가장 큰 병목 지점은 파이썬 네이티브 `for` 루프 자체가 가진 인터프리터 오버헤드입니다. N이 수천 단위로 커지면 엄청난 지연이 발생합니다.
해결 방안으로는 저수준 언어(C/C++)로 연산부를 포팅하거나, 하드웨어 SIMD 명령어 및 병렬 처리를 적극 활용하는 벡터화(Vectorization) 라이브러리인 NumPy를 도입하여 파이썬 인터프리터를 우회하는 방법이 있습니다.

## 4. FAIL 케이스 원인 분석 리포트 (평가기준 9)
`data.json` 테스트 실행 시 발생하는 FAIL에 대한 원인 분석 결과입니다:
1. **패턴 크기 불일치 (`size_5_mismatch`)**:
   입력 패턴의 크기가 예상된 필터의 크기(NxN)와 맞지 않는 경우, 시뮬레이터가 연산 전 스키마를 검증하여 인덱스 오류 발생을 막고 즉각적으로 `FAIL` 처리 및 원인을 출력합니다.
2. **동점 처리로 인한 기대값 불일치 (`size_13_1`)**:
   두 필터에 대한 MAC 점수가 부동소수점 Epsilon 허용오차 내에서 동일하게 산출될 경우 판정은 `UNDECIDED`가 됩니다. 하지만 `expected` 항목에 특정 값이 명시되어 있기 때문에, 엄격한 테스트 기준 하에 이는 실패(`FAIL`)로 집계됩니다. 알고리즘은 의도대로 정확하게 작동한 것입니다.
